# Day 01 — coding — two-sum

This is a worked example showing the expected shape of a session file. Use
it as a reference for format, not as a real transcript to copy.

## Problem / Prompt
Given an array of integers `nums` and an integer `target`, return the
indices of the two numbers that add up to `target`. Assume exactly one
solution exists, and you may not use the same element twice.

## Transcript

**Me (Clarify):** So I'm given an unsorted array of integers and a target
sum, and I need to return the indices — not the values — of the two
elements that add up to the target. A couple of questions: can the array
have negative numbers or duplicates? And is it guaranteed exactly one valid
pair exists, or could there be multiple and I return any one?

**Interviewer:** Negatives are possible, duplicates are possible, and yes —
assume exactly one valid answer.

**Me (Lay out):** Okay. The brute-force way is to check every pair, which
is O(n squared) time and O(1) space — for each element, scan the rest of
the array for a complement. That works but doesn't scale well for large n.
A better approach: as I iterate once through the array, I keep a hash map
of value-to-index for everything I've already seen. At each element, I
check if `target minus current value` is already in the map — if so, I've
found my pair. That's O(n) time and O(n) space. Given there's no mention of
memory being constrained, I'd like to go with the hash map approach — does
that sound reasonable?

**Interviewer:** Yes, go ahead.

**Me (Execute):** I'll set up an empty dictionary, `seen`, mapping value to
index. Then I loop through `nums` with the index. At each step I compute
`complement = target - nums[i]`. If `complement` is already a key in
`seen`, I return `[seen[complement], i]` immediately. Otherwise, I add the
current value and index to `seen` and continue. Since the problem
guarantees exactly one answer exists, I don't need a fallback return value.

**Interviewer:** What if the same value appears twice and together they're
the answer?

**Me (Assess):** Good case to check — say `nums = [3, 3]`, `target = 6`. On
index 0, complement is 3, not in `seen` yet, so I add `3 → 0`. On index 1,
complement is 3 again, and now it *is* in `seen` — pointing at index 0. So
I'd return `[0, 1]`, which is correct since I check `seen` before adding
the current element, not after. That ordering is exactly why this works
even with duplicate values.

**Me (Refine):** So overall this is O(n) time, O(n) space, single pass. The
trade-off I made was space for time versus the brute force — worth it here
since nothing suggested memory was tight. If it did matter, and the array
were sorted or could be sorted, a two-pointer approach would get us to
O(1) space at the cost of O(n log n) for the sort. Summary: single-pass
hash map, return the first valid pair found.

## Claude's Feedback

### Rubric (1-5 each)
- Clarify: 4 — good questions on duplicates/negatives, didn't ask about
  input size, which mattered less here but is a habit worth keeping.
- Lay out: 5 — stated brute force, complexity, and got explicit buy-in
  before coding.
- Execute: 4 — narrated the setup and the check-before-add order, but the
  initial pass didn't explain *why* checking before adding matters until
  prompted by the interviewer's question.
- Assess: 4 — walked the duplicate case correctly and explained the
  ordering; didn't proactively check the "no solution" case even though
  the problem guarantees one exists (worth a one-line acknowledgment).
- Refine: 5 — gave complexity, a real alternative (two-pointer + sort),
  and a tight one-sentence summary.

### Flags
- Hedge-stacking: absent
- Signposting: present — phase transitions were named consistently.
- Silent debugging: absent

### Specific notes
1. State the check-before-add invariant proactively in Execute, before
   being asked — that's exactly the kind of subtlety interviewers are
   listening for you to surface unprompted.
2. In Assess, briefly acknowledge the "no valid pair" case even when the
   problem guarantees one exists — one sentence ("if no solution existed,
   this would fall through the loop without returning — the problem
   guarantees that won't happen") shows you thought about it.
3. In Clarify, add input-size range to your default question list even
   for easy problems — it's a one-line question and costs nothing.

### Model verbal answer
"I'll check the complement against what I've already seen *before* I add
the current element — that ordering matters, because it's what lets a
repeated value like `[3,3]` still resolve correctly without accidentally
matching an element against itself."
